# 単一ファイル向けカスタマイズレビュー観点（javaサンプル）

このプロンプトでは 1 回の実行につき 1 ファイルのみをレビュー対象とします。
Web アプリの複雑な Controller や大規模テストクラスは扱わず、演習問題のようなシンプルなコードを対象に以下の観点を適用してください。

## Java 演習

### `Task01PrintDate.java`
- `LocalDate`/`DateTimeFormatter` を使い、現在日付と指定日付との差分が正しく計算されていること。
- 日本語表示が必要な場合は `Locale.JAPAN` を明示的に指定していること。
- 標準出力にフォーマット済み文字列のみを出し、デバッグ用出力を残していないこと。

### `Task02ArrayStats.java`
- `int[]` などの入力配列が空の場合のハンドリング（早期 return, 例外など）があること。
- 平均値や最大値の計算でオーバーフローを避けるため `long` あるいは `double` を用いていること。
- `for` ループ・拡張 `for` ループのどちらか一方に統一し、不要な重複処理がないこと。

### `Task03SimpleGame.java`
- `Scanner` や `BufferedReader` のクローズ処理が `try-with-resources` で書かれていること。
- 乱数の範囲（例: 1〜6）は要件通りかつ複数回のプレイで再利用される `Random` インスタンスを保持していること。
- 入力値チェック（数字以外が入力された場合など）が実装されていること。

### `ENSYU3_3.java`
- KeySet、ないし、EntrySetなどのコレクションクラスを使えていること。
- 表示順の組み合わせが正しいこと。

### `ENSYU4_1.java`
- 仮型引数が使えていること。

### `ENSYU4_2.java`
- 列挙型が使えていること。
- 必要施工回数未満ではnullが返ってくること。
- 必要施工回数で金庫に格納されたインスタンスが返ってくること。
- テストコードでif分岐を作っていないこと。。

### `TSUIKA_ENSYU1.java`
- 例外をthrowするのではなく、例外が発生する処理が記述されていること。
- try-catchで正常終了すること。
- printStackTraceで詳細情報が出力されていること。

### `TSUIKA_ENSYU3*.java`
- 固定値による行分けや繰り返しがされていないこと。
- 入力ファイルの中身の文字数、行数が変わっても正しく動くこと。
- 正しい例外処理が行われていること。
- 例外が発生しても必ずclose()されること。
- 2通り解答されること。
  ①try-with-resources文
  ②finallyブロックでtry-catchも記述してclose()

### `TSUIKA_ENSYU4.java`
- getter/setterは省略しないこと。
- Comparable,Cloneableインタフェースが実装されていること。
- equals()が書名と発行日を用いた条件でオーバーライドされていること。
- hashcode()が書名と発行日を用いた計算でオーバーライドされていること。
- compareTo()が出版日の条件でオーバーライドされていること。
- DateはcompareTo()を持っているので短く記述可能。
- 深いコピーになっていること（特に発行日に注意）。
- コピー元の発行日に別の日付をsetTimeしてもコピー元とコピー先の発行日が異なっていること。

### `TSUIKA_ENSYU5.java`
- 題意の文字列にマッチすること。
- String.matches()やPattern.matches()を使って、正規表現のチェックが行われていること。
- 題意の文字列以外の文字列にマッチしないこと。
